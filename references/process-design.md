# Process Design — 협상 프로세스 설계

## Single-Text Procedure (Raiffa) — 단일 문안 절차

복잡한 다자 협상이나 교착 상황에서 사용하는 구조화된 프로세스.

### 메커니즘

```
Step 1: Mediator (또는 협상 주도자)가 초안 작성
        ↓
Step 2: All Parties 검토 + 의견 피드백
        ↓
Step 3: Mediator가 의견 반영하여 개정
        ↓
Step 4: 반복 (일반적으로 3~5 라운드)
        ↓
Step 5: "Never Revert" 원칙: 이전 버전으로 돌아가지 않음
        → 한 방향 진행만 (Forward Only)
```

### 특징

| 특징 | 의미 | 효과 |
|------|------|------|
| **Never Revert** | 합의한 항목은 재협상 금지. 새로운 제안만 가능 | 교착 방지. 진행 속도 가속 |
| **Neutral Drafter** | Mediator가 중립자여야 함 (한쪽에 편향 금지) | 신뢰 확보. 합리성 강화 |
| **Opaque Process** | 누구 의견이 반영되었는지 명확히 하지 않음 | 각 측이 "우리 의견이 반영됐다"고 느낌 → 소유권 확보 |
| **Incremental Change** | 매 라운드 작은 개정만 (급격한 변화 금지) | 심리적 거부감 감소 |

### 적용 시기

- 다자 협상 (3명 이상)
- 복잡한 계약서 (수십 개 조항)
- 이전 협상 실패 후 재시도
- 상대 신뢰도 낮을 때 (제3자 mediator 신뢰 활용)

### 한계

- Mediator의 중립성이 의심되면 전략 무너짐
- 시간이 오래 걸림 (3~5 라운드)
- 초안 작성자의 숨은 영향력 있음 (프레이밍의 힘)

## Contingent Contracts — 조건부 계약

협상 당사자들이 미래 발생 사건에 대해 서로 다른 예측을 가지고 있을 때, 예측을 "베팅"으로 구조화.

### 핵심 아이디어

```
판매자 주장: "우리 제품의 고장률은 연 2%입니다"
구매자 의심: "실제로 5%일 거야"

일반 협상: "정확한 고장률에 대해 싸움" → 교착

Contingent Contract:
"만약 실제 고장률이 2% 이하면, 구매자가 추가 $50만 지불
 만약 고장률이 5% 이상이면, 판매자가 $100만 환급"

→ 판매자: "우리가 맞다고 자신 있으면 베팅 동의"
→ 구매자: "확신 없으면 베팅 거부"
→ 정보 비대칭 극복
```

### 적용 구조

| 단계 | 실행 | 예 |
|------|------|------|
| **1. 예측 차이 명확화** | "판매자는 고장률 2%, 구매자는 5%라고 예측" | 차이 구체화 |
| **2. 베팅 설정** | "기준: 연 평균 고장률. 측정: 독립 기관" | 객관성 확보 |
| **3. 보상·손실 설정** | "Target에서 1%p 차이마다 $25만 이동" | 동등한 리스크 분배 |
| **4. 계약서에 명시** | "조건: 측정 가능, 독립 검증, 분쟁 시 중재" | 법적 구속력 |
| **5. 이행** | "매년 말 결과 확인 → 자동 조정" | 자동화. 분쟁 최소화 |

### Contingent Contract의 종류

| 종류 | 메커니즘 | 사례 |
|------|---------|------|
| **Earnouts** (실적 연동) | 인수 대금의 일부를 실적에 따라 지불 | M&A: "기본 $500만 + 2년 EBITDA $100만 이상 시 $200만 추가" |
| **Performance Bonds** | 성과 미달 시 환급 | "납기 지연 시 1일마다 $5,000 환급" |
| **Royalty Contracts** | 매출에 따라 지불 비율 변동 | "연 매출 $100만 이상 시 로열티 5%, 그 이상은 3%" |
| **Indemnification** | 리스크 발생 시 보상 | "숨겨진 채무 발견 시 판매자가 $X 배상" |
| **Escrow** | 일부 금액을 제3자 보관. 조건 충족 시 지급 | "계약금 $100만을 은행이 보관. 납기 완료 후 지급" |

### 효과

| 효과 | 설명 |
|------|------|
| **정보 비대칭 극복** | 거짓 예측은 베팅 비용. 거짓말 동기 약화 |
| **신뢰도 향상** | "판매자가 자신 있으니까 베팅에 동의했다" → 신뢰 신호 |
| **협상 수렴 가속** | 불확실성을 "베팅"으로 구조화 → 합의 용이 |
| **마찰 감소** | 미래 분쟁을 사전에 정량화 → 해석 여지 없음 |

### 주의

- **베팅 설정이 공정해야 함**: 한쪽에 유리하면 다른 쪽이 거부
- **측정 가능해야 함**: "고객 만족도"처럼 주관적 지표는 분쟁 야기
- **검증 중립성**: 누가 측정하는가? (독립 기관 필수)

## Post-Settlement Settlement (PSS) — 합의 후 재협상

초기 합의 후, "실제로 우리 둘 다 더 선호하는 대안이 있나?"를 탐색.

### 메커니즘

```
Phase 1: 초기 합의 도달 (A안)
         "이 안으로 합의합니다"

Phase 2: 심호흡. 초기 압박 제거

Phase 3: "혹시 이것보다 더 좋은 대안이 있지 않을까요?"라는 개방형 탐색
         (초기 합의가 "default"이므로 협상 압박 없음)

Phase 4: 새로운 대안 발견 (B안)
         양측이 A안보다 선호하면, B안으로 변경

Phase 5: 최종 합의 (B안)
```

### 효과

| 효과 | 설명 |
|------|------|
| **파이 확대** | 초기 협상에서 놓친 창의적 옵션 발견 |
| **만족도 향상** | "가능한 최고"가 아닌 "실제로 우리가 가장 선호하는" 안 도달 |
| **관계 개선** | 압박 없는 환경에서 함께 생각 → 협력 강화 |
| **교착 해결** | "초기 합의가 있으니까 안 된다고 거부할 수 없지만, 더 나은 방법 탐색 가능" |

### 한계

- 초기 합의가 "충분히 좋아야" PSS 동의 가능 (나쁜 안이면 "이대로 가자"라고 고집)
- 정교한 당사자 필요 (naive buyer/seller는 PSS 이해 어려움)
- 시간 소요 (새로운 라운드)

### 실전 팁

"PSS"라는 용어 쓰지 말 것. 그냥 "혹시 더 나은 구조가 있나 함께 생각해보면?"이라고 자연스럽게 제안.

## Logrolling & Value Creation — 로그롤링과 가치 창출

### Logrolling (다중 쟁점 번들)

서로 다른 우선순위를 가진 여러 쟁점을 "거래"하는 기법.

| 예 | 우리 우선순위 | 상대 우선순위 | Logrolling |
|---|----|----|----|
| 계약 가격 + 기간 | 낮음 (유연) | 높음 (중요) | "기간 3년 고정" → "가격 $70만으로 낮춤" |
| 가격 + 지급 방식 | 높음 (현금 필요) | 낮음 | "선금 없음, 실적 연동" → "가격 $80만 수용" |
| 가격 + A/S 기간 | 낮음 | 높음 | "A/S 5년" → "가격 타협 가능" |

### 전제 조건

1. **다중 쟁점 필수**: 단일 쟁점 (가격만)이면 logrolling 불가
2. **서로 다른 우선순위**: "우리는 가격 중시, 상대는 기간 중시" → logrolling 가능
3. **정보 공개**: 서로의 우선순위를 알아야 번들 가능

### Logrolling 프로토콜

```
Step 1: 쟁점 나열 (최소 3개 이상)
        우리: 가격, 기간, A/S
        상대: 기술 지원, 독점성, 선금

Step 2: 각 측의 우선순위 파악 (명시하지 않고 "질문"으로)
        우리: "기간이 가장 중요한 부분이 뭔가요?"
        상대: "기술 지원을 충실히 해주세요"

Step 3: "Simultaneous Package" 제시
        우리: "기간 3년 + 기술 지원 월 1회 → 가격 $70만"
        상대: 검토

Step 4: 조정 (상대의 우선순위 추가)
        상대: "독점성은 빼고 싶어요"
        우리: "그럼 가격은 $75만으로"

Step 5: 최종 합의
```

### 주의

- **Sequential offer 금지**: "먼저 이것에 동의하고, 다음에 저것" → logrolling 아님
- **Simultaneous 강조**: 전체 패키지를 동시에 제시 → 시너지 인식

## Value Creation 5가지 방법

파이를 확대하는 5가지 구조적 방법.

| 방법 | 정의 | 예 |
|------|------|------|
| **1. Adding Variables** (변수 추가) | 새로운 쟁점 추가 | 가격만 있던 협상 → 기간, A/S, 기술 지원 추가 |
| **2. Reformulating Issues** (쟁점 재구조화) | 문제를 다르게 정의 | "우리가 납기 일정을 맞출 수 없다" ↔ "당신은 정확히 언제부터 필요한가?" (유연성 발견) |
| **3. Temporal Restructuring** (시간 구조 변경) | 지급·납기 시간 조정 | "한 번에 $100만 지불" → "월 $10만 × 10개월" (구매자 현금흐름 개선) |
| **4. Conditional Terms** (조건부 조건) | 미래 사건에 따라 변동 | "실적 목표 달성 시 가격 인상 2%" (리스크 공유) |
| **5. Non-Monetary Substitution** (비금전 대체) | 금전 아닌 가치 제공 | "가격은 높지만, CEO 직접 컨설팅" 또는 "우선 거래권" (상대가 중시하는 가치) |

### Value Creation 체크리스트

협상 중간중간, "파이를 더 키울 수 있는가?"를 체크:

```
1. 추가할 쟁점이 더 있나? (변수 추가)
   → 기술 지원, A/S, 교육, 컨설팅 등

2. 같은 문제를 다르게 정의할 수 있나? (재구조화)
   → "가격이 높다" = "우리의 cash flow 제약"
   → "납기가 짧다" = "정확히 언제 필요한가?"

3. 시간을 다르게 구조화할 수 있나? (시간 조정)
   → 선금 → 분할 지불
   → 선 납기 → 단계 납기

4. 미래 불확실성을 베팅으로? (조건부)
   → "고장률이 낮으면 보너스"

5. 금전이 아닌 것은? (비금전 대체)
   → 기술 지원, 브랜드 연결, 독점성
```

## Deadlock Breaking Protocol — 교착 해결 5차원 매트릭스

교착이 발생했을 때, 원인을 정확히 진단하고 대응.

### 5-Layer × 5-Pattern Matrix

| Layer/Pattern | Position 고착 | 정보 비대칭 | 감정 에스컬 | 체면·권위 | 약속 신뢰도 |
|---|---|---|---|---|---|
| **L1 Interest** | 이해관계 미탐색 | 숨겨진 우선순위 | - | - | - |
| **L2 Structure** | ZOPA 부재 | 불투명한 대안 | - | - | - |
| **L3 Psychology** | 고정된 앵커 | 의도 의심 | 손실 고착 프레이밍 | 지위 경쟁 | - |
| **L4 Relationship** | 신뢰 부족 | 신뢰 결핍 | 관계 악화 | 문화적 권력 | 신뢰도 약함 |
| **L5 Execution** | - | - | - | - | 약속 실행 메커니즘 부재 |

### 진단 → 처방

| 진단 신호 | 추천 기법 | 기대 효과 |
|-----------|---------|---------|
| "왜 그 조건이 필요한가"에 답 못 함 (Position 고착) | Interest 깊이 탐색 (5단계 "왜?") | Position의 근거 찾기 → 협상 재개 |
| "우리와 상대의 대안이 뭔지 불명확" | BATNA·ZOPA 명시 (양측) | 합의 가능 범위 가시화 |
| "상대가 자꾸 의심한다" (반응 평가절하) | 제3자 중재자 도입 | 중립자의 제시 → 의심 감소 |
| "감정이 고착되어 논리가 먹히지 않음" | 쿨다운 + 감정 해제 (Ury) | 차분한 상태에서 재협상 |
| "상대가 최종 결정을 못 한다" (주인-대리인) | 상급자 개입 요청 + 권한 명확화 | 실제 결정권자와 협상 |
| "계약서를 만들어도 이행을 안 함" | Commitment Device 강화 (공개 선언, 제3자 감시, 구속력 있는 조건) | 이행 신뢰도 향상 |

## 협상 전·중·후 체크리스트 (Process Design 관점)

### 협상 전 (Setup Phase)

```
☐ 참석자 명확화 (우리 팀, 상대팀, 제3자)
☐ 회의 장소·시간 결정 (누가 주도하는가?)
☐ 의제(Agenda) 설정 (순서, 예상 시간)
☐ 양측 BATNA 사전 파악
☐ 상대의 문화 프로필 파악 (Hofstede/Meyer)
☐ 우리 측 내부 합의 (선금 limit, 최종 한계점)
☐ 심리 스크리닝 (상대의 앵커링/손실회피 약점)
```

### 협상 중 (Negotiation Phase)

```
☐ First offer 전술 (누가 앵커를 제시하는가?)
☐ 5-Layer 스크리닝 (●/○/- 판정)
☐ 상대의 Interest 탐색 (5단계 "왜?")
☐ 옵션 창출 (단일 쟁점 → 다중 쟁점으로)
☐ Logrolling 기회 탐색 (서로 다른 우선순위)
☐ 감정 관리 (상대의 공격에 감정적 반응 금지)
☐ 정보 공개 전술 (우리는 투명, 상대는 유도)
```

### 협상 후 (Implementation Phase)

```
☐ 합의사항 문서화 (MOU or Contract)
☐ 이행 감시 메커니즘 (누가? 언제? 어떻게?)
☐ 조건부 계약 검토 (미래 불확실성 대비)
☐ 양측 내부 승인 (우리 상급자, 상대 상급자)
☐ Contingency 사전 정의 ("만약 X가 발생하면?")
☐ 정기 리뷰 일정 (월/분기별 check-in)
☐ Escalation path (분쟁 발생 시 해결 절차)
```
